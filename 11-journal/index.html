    Chapter 11 — The Journal as Mental Model¶

    You will not write pipeline code. You will read three bundles with planted bugs and find them.

    Prerequisites¶

    • Chapters 0-10. This is a consolidation chapter.
    • uv sync --group dev. No API keys needed — everything is offline against pre-recorded bundles.

    Minimum to skip the ladder: chapter 2 alone (the first chapter that writes a bundle). The investigation flow doesn't depend on you having built the pipeline, just on knowing what the records mean.

    Read-only by design¶

    This is the one chapter where you do not produce audio. There's nothing for a microphone to do. The "runnable at every checkpoint" rule bends here because the chapter's whole point is reading bundles, not making them. The fixtures under bundles/ are checked in so you can work offline, on a plane, without API keys, without a mic — pick the constraint that fits.

    If you do want to scrub a live bundle: any chapter from 2 onward dumps one to its own runs/ directory. Chapter 9c (docs/teaching/09-interruption/runs/) is the most interesting — barge-in turns have the richest record set. Open one with investigate.py in this chapter once you've finished the three planted-bug investigations below.

    Diff from chapter 10¶

    • Added: investigate.py — a tiny CLI bundle browser; bundles/bug_*.bundle — three planted-failure fixtures checked in (unlike the gitignored runs/* from earlier chapters); generate_bundles.py that builds those fixtures; an explicit walkthrough of the JournalView query surface (filter_by_stage, filter_by_turn, lookup_by_sequence).
    • Removed: the entire live pipeline. This chapter writes no audio.

    The journal is not logs¶

    • Logs are unstructured prose. You grep them. Your search depends on your choice of words.
    • The journal is a strictly ordered, typed, per-turn event stream. You query it: by stage, by turn, by sequence. Every record has a stable schema.

    Every runnable chapter from chapter 2 onward has been dumping bundles to runs/. This chapter teaches you what those bundles are for.

    Bundle safety. Bundles are ZIP/JSON archives that EasyCat validates for structure and path traversal, but they are not signed or tamper-evident. Only open bundles you generated yourself or got from a source you trust. (The ones checked in under bundles/ are generated by the companion scripts in this repo.)

    The three bugs¶

    The bundles live in bundles/*.bundle — checked in, unlike the gitignored runs/* from other chapters, because this chapter can't work without them. They were generated by a companion script:

    uv run python docs/teaching/11-journal/generate_bundles.py

    Each bundle records a session with a specific, planted failure. Your job is to find the failure using only the journal.

    Your tool¶

    One tiny script:

    uv run python docs/teaching/11-journal/investigate.py \
    docs/teaching/11-journal/bundles/bug_01_empty_final.bundle

# Filter by stage or by record name:
uv run python docs/teaching/11-journal/investigate.py \
    docs/teaching/11-journal/bundles/bug_02_tts_stutter.bundle --stage tts

uv run python docs/teaching/11-journal/investigate.py \
    docs/teaching/11-journal/bundles/bug_03_ghost_interruption.bundle --name interruption.start

    Under the hood it is:

    from easycat.debug.testing import load_bundle
b = load_bundle(path)
for r in b.records():
    ...

    You can also use the JournalView API for more structured queries — filter_by_stage, filter_by_turn, lookup_by_sequence.

    Investigation 1 — guided¶

    Open bug_01_empty_final.bundle. Work the evidence:

    1. turn.started at t_ms=1,000,000. User is expected to speak.
    2. stt.final at t_ms=1,001,100. Look at data["text"].

    What do you see? An empty string. The turn committed a final but there was no transcript.

    Now look for the agent:

    ... --stage agent

    Nothing. The agent stage never ran.

    Write down your hypothesis — why did STT commit an empty final? — in a single sentence before opening solutions.md.

    Investigation 2 — semi-guided¶

    Open bug_02_tts_stutter.bundle. The user hears stuttering — fine, gap, fine — between sentences. Hypotheses to walk:

    • Sentence splitter wrong? Look at the stage.tts.execute records. How many? Three? That's correct for a three-sentence reply.
    • Agent stalls? agent.first_token at t_ms=2,001,400 — fast. No other agent records; probably fine.
    • TTS slow? Compare elapsed_ms on each stage.tts.execute. One is 5× the others. Why? The records between them — ws.reconnect.* — tell you.
    uv run python docs/teaching/11-journal/investigate.py \
    docs/teaching/11-journal/bundles/bug_02_tts_stutter.bundle --stage tts

    Investigation 3 — unguided¶

    Open bug_03_ghost_interruption.bundle. The bot thinks it was interrupted but the user never spoke. Find the journal evidence for:

    • Which record told the coordinator a barge-in happened?
    • What is missing from around that record that would confirm the interruption was a real user?
    • What setting in the journal explains why the bot self-triggered?

    Write your hypothesis in one sentence before peeking at solutions.md.

    Why the journal has to be structured¶

    Three properties make this kind of investigation tractable:

    1. Strict ordering by sequence number. You can say "what happened after sequence 14" and mean it.
    2. Typed fields. data["text"] is always a string; data["t_ms"] is always a float. No regex parsing.
    3. Causal grouping by turn_id. In multi-turn bundles, all records for one turn share the same id and can be isolated.

    The teaching shape vs. the production shape¶

    Chapters 2-10 and the fixtures here emit composite events of the form stage.<name>.execute with a single elapsed_ms field — one record per stage span. This is deliberate: the query layer stays at the surface and the reader can read the bundle top-to-bottom.

    The production journal in src/easycat/runtime/ emits paired records: stage_start at the beginning of a span, a matching stage_complete at the end, with artifact refs and error blobs in between. You reconstruct a span by matching start/complete on op_id. Paired records buy you partial-span visibility on crashes and concurrent-span overlap that single events can't express. Read the production records after you've internalised the teaching ones.

    Also be aware: turn.state_changed and ws.reconnect.* records in these bundles are representative synthetic records — the production pipeline emits similar signals, but chapters 2-10's own scripts do not write them. They are here so the fixtures tell a coherent story without running the whole framework.

    Try breaking it¶

    Pick any bundle from your runs/ directories — chapter 9 is good — and write three hypotheses for what could go wrong on a flaky day. For each hypothesis, name the record you'd query first.

    Then plant your own bug in chapter 9c's code. Dump the bundle. Can a classmate (or your future self) find it?
