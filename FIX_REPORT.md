## Repair pass — regressions fixed & fixes reverted

Several audit findings proved **wrong on closer inspection**: the verifier flagged code as dead/removable, but tests proved it was live. Those removals were reverted.

### R1-agents-streaming — green
- **Root cause:** base.py: configure_runtime was added to the @runtime_checkable ExternalAgentBridge Protocol body. Since runtime-checkable isinstance requires ALL declared methods to be present, this made isinstance(obj, ExternalAgentBridge) return False for every bridge (GenericWorkflowBridge, langchain/langgraph, etc.) and for AgentRunner / test bridge fakes that legitimately no-op it. This broke AgentRunner._is_bridge detection (so bridge-wrapping fell through to the .run() non-bridge path -> AttributeError, including the 'hanging' test_bridge_delegation_honors_configured_timeout), factory passthrough, facade, and the contract/example isinstance assertions. The audit's actual intent (per config._apply_runtime_settings) was that configure_runtime be an OPTIONAL surface probed via getattr/hasattr with a private-attribute fallback.
- **Root cause:** _streaming.py: the audit replaced the blocking stop-sentinel put with put_nowait(None) to avoid hanging in finally when a barge-in cancelled the consumer. But on CLEAN completion against the bounded (maxsize) queue, the last payload could still occupy the queue when the producer finished, so put_nowait raised QueueFull and dropped the sentinel -> the actively-draining consumer blocked forever on its final get(). Caused test_consume_agent_stream_applies_backpressure_on_bounded_queue to hang.
- **Root cause:** _turn_runner.py gated keep-alive: the gated-replay branch in _process_tts calls TurnManager.reset() with the comment 'Keep current turn alive', relying on the OLD reset() NOT cancelling the token. R5-misc's audit change made TurnManager.reset() cancel its active token before dropping it (asserted by tests/turns/test_turn_manager.py::test_reset_returns_to_idle). That cancellation killed this turn's live token while the agent stream ran concurrently, so stream_succeeded went False and AgentFinal was never emitted (test_create_session_replays_gated_audio_after_human_classification timed out). This is a cross-cluster interaction: R5's reset semantics are intentional, so the fix belongs at my call site.
- **Source fix:** src/easycat/integrations/agents/base.py: removed configure_runtime from the @runtime_checkable ExternalAgentBridge protocol body and documented it as an optional extension surface (probed via getattr in config._apply_runtime_settings). Restores isinstance() for all bridges, AgentRunner, and the contract/test fakes that no-op it. This is the cascade root-cause fix.
- **Source fix:** src/easycat/session/_streaming.py: in the finally sentinel logic, use a blocking 'await tts_queue.put(None)' on clean completion (consumer is still draining, sentinel delivery must be guaranteed) and keep the non-blocking put_nowait(None)+QueueFull-swallow path only for the cancel/error case where the TTS consumer may have been cancelled by a barge-in. Preserves both the backpressure and the no-hang-on-barge-in intents.
- **Source fix:** src/easycat/session/_turn_runner.py: added _reset_turn_manager_preserving_token() and call it (instead of TurnManager.reset()) in the gated keep-alive branch. It detaches the manager's live token before reset() so reset() cancels nothing, while the turn keeps its own token via turn.cancel_token / the captured local. Preserves R5's new reset()-cancels-token semantics for all other callers (barge-in, idle teardown).
- **Test updated (intentional change):** tests/integrations/agents/test_ws2b_interruption_and_mcp.py::TestApplyInterruptionFourStepOrder::test_four_step_method_calls_present[OpenAIAgentsBridge/PydanticAIBridge/GenericWorkflowBridge]: the four-step protocol was intentionally hoisted into the shared run_interruption_journal_protocol free function, so apply_interruption no longer directly CALLS _apply_planned_mutation (it passes it as the apply_mutation= callback). Updated the AST walk to assert apply_interruption calls _plan_interruption AND run_interruption_journal_protocol, and references _apply_planned_mutation as an attribute (the mutation callback). Preserves the test's intent (planned mutation flows through the four-step atomic ordering).
- **Test updated (intentional change):** tests/integrations/agents/test_responses_api_bridge.py::TestApplyInterruptionFourStep::test_four_step_method_calls_present: same shared-function structural update for RemoteResponsesAPIBridge.

Out of scope for a minimal, behavior-preserving pass.

The finding states 'Do not gate a release on this - there is no current defect, only a footgun for future edits.' Extracting the 385-line constructor mechanically is high-risk for init-order regressions and exceeds a surgical fix scope.
- **xc-architecture-6** — config.py (1593 LOC) conflates config dataclasses, the session factory, telephony/outbound wiring, and debugger-UI launching: Same multi-file module-split as xc-ease-of-use-5 (overlapping scope), additionally requiring _maybe_launch_debugger_ui to move next to the debugger package and re-pointing 7 internal imports of private config symbols across _logging.py, helpers.py, cli/scaffold/init.py, and session/_session.py while keeping package-root re-exports. Out of scope for a single-file tier-1 agent; the rec marks it 'lower priority than any correctness/robustness finding' and reasonable to defer.
- **xc-ease-of-use-5** — config.py is a 1.6k-line module that couples the 'super easy' EasyConfig path to the entire outbound-telephony/IVR/screening wiring: Requires extracting outbound/telephony runtime wiring (_OutboundPipelineWiring, _run_agent_once, _wire_outbound_pipeline, _create_outbound_helpers, heavy telephony imports) into a new module and re-pointing imports, plus updating the two telephony test modules that import _create_telephony_helpers from easycat.config. This is multi-file refactoring that violates the tier-1 single-file constraint; the rec itself classifies it as low-severity and 'reasonable to defer'.
- **session-core-5** — Session keeps six (not eight) test-compat shim delegates on its surface; two are fully dead with zero callers: Deleted the two genuinely dead shims with zero callers in src/tests/examples: _on_turn_ended and _execute_text_turn (no-behavior-change removal; production EventBus subscribes to self._turn_runner.* directly).
