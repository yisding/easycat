## Executive Summary

EasyCat is a mature, well-factored framework with strong observability ambitions, but the audit surfaces three recurring patterns that pull it away from its goals of *super easy to use*, *highly robust*, and *easy to understand*. First, **teardown and turn-start error paths leak resources**: the headline `record_to` flag crashes the documented `async with session:` / `run()` happy path with a `TypeError`, and a transient STT failure during pre-roll priming leaks a consumer task, an open stream, and a stuck turn — both directly contradicting "resources are always cleaned up." Second, **the journal — documented as the single source of truth — does not deliver on its durability and consistency promises**: the default SQLite backend never commits mid-session (SIGKILL loses everything), degraded mode is recorded inconsistently across backends, and the CLI's `bundles show` reports wrong duration/tool-call counts for the common exported-bundle case. Third, **pervasive duplication and boundary leaks erode understandability**: the four-step interruption protocol, WS-TTS error emission, and SQL INSERT logic are each copy-pasted 3–7 times, while `Session` (1822 LOC) and `config.py` (1593 LOC) remain god-objects that reach into collaborator privates with `# type: ignore`. The most severe, lowest-effort issues cluster in lifecycle/teardown and the runtime/journal subsystem; the architectural debt is real but mostly medium-priority, incremental refactoring rather than rewrites.

# 2. Ease of Use

All four anchors are confirmed exactly as described in the findings. The evidence is solid. Writing the section now.

## Ease-of-Use

The framework's stated north star is minimal-friction onboarding — "voice bot in three lines," sensible defaults, one obvious knob per concern. The findings below show that the *most-advertised* paths (the `record_to` recording flag, `run()`, README provider swaps, `bundles show`) are where the friction concentrates. Two are outright crashes on documented happy paths.

### Theme 1 — The documented happy path crashes (highest impact)

**1. `record_to` recording flag breaks `async with session:` / `shutdown()` with a `TypeError`** *(merges `config-public-api-0` + `xc-architecture-0`)*
`_install_record_to_hook` rebinds `session.stop = _wrapped_stop`, but `_wrapped_stop()` takes **no parameters** while the real contract is `async def stop(self, *, force: bool = False)` and `__aexit__`/`shutdown()` both call `stop(force=True)`.
- Anchors: `src/easycat/config.py:1021` (`async def _wrapped_stop() -> None`), `:1033` (the rebind), `src/easycat/session/_session.py:1128` (`__aexit__` → `stop(force=True)`).
- Why it hurts: `record_to` is the "always be recording" flag, so `run(EasyConfig.mic(record_to=..., debug='light'))` — the exact incantation for capturing failures — raises `TypeError: _wrapped_stop() got an unexpected keyword argument 'force'` at teardown, leaking transport/journal/backends and corrupting the very bundle the user asked for. `_wrapped_stop` *also* always calls `original_stop()` with default `force=False`, silently destroying the graceful-vs-force distinction. CI is green because `tests/test_record_to.py` only calls bare `stop()`/`shutdown()`, never `stop(force=True)` or `async with`.
- Severity: **high** · Effort: **trivial**
- Fix: make the wrapper signature-transparent — `async def _wrapped_stop(*, force: bool = False): try: await original_stop(force=force) finally: await _export_bundle()`. Since `shutdown()` already delegates to `stop(force=True)`, wrap `stop` alone and drop the separate `_wrapped_shutdown` + its `already_exported` dedup. Add a regression test that drives teardown via `async with session:` and gives the fake session the real `(self, *, force=False)` signature. Longer term, replace the monkeypatch with a first-class Session teardown/lifecycle callback so the signature can't drift.

**2. `run()` — the "three-line" entry point — crashes on Windows** *(`xc-ease-of-use-0`)*
`run()` unconditionally calls `loop.add_signal_handler(sig, stop_event.set)`, which raises `NotImplementedError` on Windows' default `ProactorEventLoop`.
- Anchors: `src/easycat/helpers.py:114-115`, also `:33-34` in `wait_for_shutdown_signal()`; advertised at `__init__.py:7` and `README.md:112`.
- Why it hurts: the single most-advertised onboarding call (`run(EasyConfig.mic(agent=...))`) raises before the bot ever starts, with a traceback pointing at asyncio internals, not user code. A newcomer on Windows can't get past line 3 of the README.
- Severity: **medium** · Effort: **small**
- Fix: wrap each `add_signal_handler` in `try/except NotImplementedError`; on fallback, rely on `asyncio.run` propagating `KeyboardInterrupt` from Ctrl+C (teardown still runs via `async with session:` `__aexit__`) so Windows users get a clean exit.

**3. `EasyConfig(enable_noise_reduction=True)` crashes when no backend is installed** *(`vad-noise-echo-1`)*
`create_noise_reducer()` auto-mode returns `PassthroughNoiseReducer()` (with `is_passthrough_provider = True`) when neither Krisp nor RNNoise is present, but `Session.__init__` *rejects* passthrough providers with a hard `ValueError`.
- Anchors: `src/easycat/noise_reduction.py:281`, `src/easycat/session/_session.py:223-232`, `config.py:820-824`.
- Why it hurts: explicitly enabling noise reduction on a stock machine crashes at construction instead of warning-and-continuing — the factory's graceful-degradation philosophy and the Session's strict-noop guard directly contradict each other. The `ValueError` text ("non-noop implementations for: ...") is opaque to an ease-of-use-first user.
- Severity: **medium** · Effort: **small**
- Fix: add `fallback_policy: Literal["passthrough","error"]` to `NoiseReducerConfig` mirroring `create_echo_canceller` (`echo_cancellation.py:226-234`); default `"passthrough"` logs an actionable "install `easycat[rnnoise]`" warning instead of crashing. At minimum, rewrite the `ValueError` to name the missing extra.

### Theme 2 — Debug/triage tooling reports wrong data

**4. `easycat bundles show`/`inspect` always shows `duration=n/a` and `tool_calls=0` for exported ZIP bundles** *(`debug-validation-cli-0`)*
`_summarise_bundle` reads top-level `record.get("wall_ns")` and counts `name == "ToolCallStarted"`, but exported ZIP records nest the timestamp under `timing.wall_ns` and the journal sink records tool calls under snake_case `"tool_call_started"`.
- Anchors: `src/easycat/cli/debug/bundles.py:88` (`wall_ns`), `:95` (`"ToolCallStarted"`); export shape at `debug/export.py:127`, sink name at `session/_journal_sink.py:101`.
- Why it hurts: `bundles show` is advertised as the fastest way to triage "what got recorded?" — and its two headline fields (duration, tool-call count, exactly what a developer scans first) are silently wrong for *every* bundle produced by `export_debug_bundle(...)`. The debugger server's `_summarise_turns` reads `timing.wall_ns` correctly, so the two surfaces disagree.
- Severity: **high** · Effort: **small**
- Fix: fall back to `record["timing"]["wall_ns"]` when top-level `wall_ns` is absent, and match `"tool_call_started"`. Replace the misleading fixture in `tests/cli/test_bundles.py:101` with real export-shaped records (or drive it through `export_debug_bundle`), asserting non-null duration and `tool_calls == 1`.

### Theme 4 — Docs, examples, and quick-helpers contradict the code

- Why it hurts: a user swapping to Deepgram is told to install (and `doctor` reinforces) an SDK that is never used — pure install tax and a docs/code contradiction that makes the docs look unreliable.
- Severity: **low** · Effort: **trivial**
- Fix: align with the already-correct `cartesia=[]` marker-only pattern — set `deepgram=[]`/`elevenlabs=[]` in `pyproject.toml`, drop `--extra deepgram`/`elevenlabs` from example docstrings, and remove the dead `doctor.py:88-89` probes. **8. Per-provider examples hand-wire config classes instead of the one-token string shortcut** *(`xc-ease-of-use-3`)*
README and `openai_agents_voice.py:23` promise swaps as one token (`stt="deepgram/nova-2"`), but `deepgram_voice.py` imports `DeepgramTTSConfig` and manually sets `api_key`/`model`/`sample_rate=16000` plus a matching transport — duplicating the work `auto_align_tts_output_to_transport` already does.
- Anchors: `examples/deepgram_voice.py:26`, `examples/combined_providers.py:31`; auto-align at `config.py:529` / `:168`.
- Why it hurts: the examples are the second thing a newcomer reads and they contradict the "change one token" promise, leaking provider-internal config and failing to demonstrate the auto-align feature that exists precisely to make string shortcuts work against the 24 kHz default transport.
- Severity: **low** · Effort: **small**
- Fix: reduce the examples to the documented `stt=`/`tts=` string form on the default transport; optionally keep *one* explicit-config example as a labeled escape hatch (e.g. telephony 8K/16K).

**9. `quick._resolve_api_key` is missing `cartesia` and defaults every unknown provider to `OPENAI_API_KEY`** *(`xc-ease-of-use-1`)*

- Anchors: `src/easycat/quick.py:37` (`.get(provider, "OPENAI_API_KEY")`), `:26`; registered at `stt/factory.py:43`, `tts/factory.py:40`.
- Why it hurts: `transcribe_file(path, provider="cartesia")` / `speak(..., provider="cartesia")` (teaching helpers used throughout `docs/teaching/`) resolve the wrong env var — either a misleading "set $OPENAI_API_KEY" error, or feeding an OpenAI key to Cartesia and failing later with an opaque 401. This is a *third* drifted copy of the provider→env-var map.
- Severity: **low** · Effort: **trivial**
- Fix: delete the local map and resolve from the drift-guarded factory catalogs (e.g. an `env_var_for(name)` accessor / `_PROVIDER_ENV_VAR`), reusing `validate_name` for a clear unknown-provider error. Add a test asserting cartesia resolves `CARTESIA_API_KEY`.

**10. AGENTS.md documents removed/private teardown methods and wrong `stop`/`shutdown` semantics** *(`xc-understandability-0`)*
AGENTS.md:52-54 references `Session.destroy()` / `Session.close()` (which exist only as private `_destroy`/`_close`) and describes `stop()`/`shutdown()` as identical full-teardown calls, never mentioning the `force` parameter.

### Theme 5 — Inconsistent sibling APIs (harder to learn, footguns when swapping)

### Theme 7 — Module-altitude debt (a newcomer's first file is buried in advanced wiring)

**17. `config.py` (~1.6k LOC) couples the "super easy" `EasyConfig` path to the full telephony/transport/provider stack** *(merges `config-public-api-1` + `xc-ease-of-use-5` + `xc-architecture-6`)*

- Anchors: `src/easycat/config.py:44` (eager imports), `:1261`/`:1324`/`:1425` (outbound wiring), `:795` (`create_session`); lazy-load contract at `__init__.py:14`/`:37`.

Add a regression test asserting `easycat.EasyConfig` does **not** import `easycat.telephony.*`.

**Cross-cutting pattern:** the two highest-severity items (#1, #4) share a root cause — *tests assert the easy shape but never the real one* (`tests/test_record_to.py` never calls `stop(force=True)` or `async with`; `tests/cli/test_bundles.py` uses a hand-built fixture that doesn't match the real export serialization). # 3. Robustness

## Robustness

This section audits the "robustness" north-star: resources are always cleaned up, a turn is never silently corrupted, and the journal faithfully records what happened — especially on transient-failure and barge-in paths. Findings are grouped by theme and ordered by impact.

### 1. Turn-start failure leaks STT resources and strands the turn (HIGH)

Two findings (`session-core-0`, `xc-robustness-0`) describe the **same defect from different angles** and should be fixed together: the `except` block in `TurnRunner.on_turn_started` does too little when pre-roll priming or `start_stream()`/`start_event_loop()` fails.

- **What** — The order is `start_stream()` → `mark_active()` → `start_event_loop(turn)` (spawns the `_consume` task) → prime pre-roll. If any of these raise, the handler only does `await self._emit(Error(...)); self._stt.mark_inactive(); return`. `mark_inactive()` just sets `self._active = False` (`_stt_committer.py:102`) — it never cancels the already-started consumer task, never `end_stream()`s the open provider stream, never resets the turn pointer set at `_turn_runner.py:178`, and never moves the TurnManager out of `USER_SPEAKING`.
- **Anchors** — `src/easycat/session/_turn_runner.py:178,188,190,196`; `src/easycat/session/_stt_committer.py:102`.
- **Why it hurts robustness** — A single transient provider error at turn start (the reconnect window — exactly when this fires) leaks a running asyncio consumer and an open websocket for the rest of the session. The orphaned consumer keeps emitting `STTPartial`/`STTFinal` and can resolve futures on *later* turns; meanwhile the live turn pointer + stuck FSM means the user's whole utterance is silently dropped (dead air until the silence timeout), with no spoken error. This violates both "resources are always cleaned up" and "a turn is never silently corrupted."
- **Severity** high · **Effort** small.
- **Fix** — In the `except` path, call the existing `STTCommitter.cancel(turn)` (it already does `end_stream() → _active=False → cancel/await _stt_task → resolve_pending`, `_stt_committer.py:347-365`) and then `self._reset_turn_state()`. The cleaner structural fix is to **move `start_event_loop` below the pre-roll loop** so the consumer task only exists once both `start_stream()` and priming have succeeded, eliminating the orphan window entirely. Add a regression test (`ScriptedSTT.fail_on_send`) asserting the turn pointer is reset, the FSM returns toward IDLE, and no STT task is left pending.

### 2. `record_to` monkeypatch crashes the documented teardown path (HIGH)

Findings `config-public-api-0` and `xc-architecture-0` are the **same bug**: `_install_record_to_hook` replaces `session.stop` with a zero-arg `async def _wrapped_stop()`.

- **What** — The real contract is `async def stop(self, *, force: bool = False)`, and `__aexit__` calls `await self.stop(force=True)` (`_session.py:1128`) while `shutdown()` aliases `stop(force=True)`. After any `EasyConfig(record_to=...)`, `async with session:` / `easycat.run(...)` raise `TypeError: _wrapped_stop() got an unexpected keyword argument 'force'`. Even called positionally, the wrapper always invokes graceful `original_stop()`, silently destroying the graceful-vs-force distinction.
- **Anchors** — `src/easycat/config.py:1021,1033`; `src/easycat/session/_session.py:1128,1276`; `tests/test_record_to.py`.
- **Why it hurts robustness** — `record_to` is the "always be recording" flag and `async with session:` is the documented primary teardown idiom — so the exact users who enabled recording to capture failures crash on exit, leaking transport/journal/backends and corrupting the very bundle they asked for. The bug is invisible to CI because the existing tests only call `stop()`/`shutdown()` with no args and a fake whose `stop` also takes no `force`.
- **Severity** high · **Effort** trivial.
- **Fix** — Make the wrapper signature-transparent: `async def _wrapped_stop(*, force: bool = False): try: await original_stop(force=force) finally: await _export_bundle()`. Since `shutdown()` already delegates to `stop(force=True)`, wrapping `stop` alone suffices — drop the separate `_wrapped_shutdown` and its dedup. Better long-term: replace the monkeypatch with a first-class Session teardown hook so signatures can't drift. Regression test must give the fake `stop` the real `(self, *, force=False)` signature **and** drive teardown via `async with session:`.

### 3. Journal does not survive the crash it is built to debug (HIGH)

- **`runtime-observability-0`** — `SqliteJournal.__init__` opens one transaction (`BEGIN`, `journal.py:829`) that `_do_append` never commits; the only `COMMIT`s are in `close()`/`finalize()`/`flush()`, and `flush()` has **no production caller**. Under WAL mode, a SIGKILL/OOM/segfault mid-session leaves every INSERT as uncommitted WAL frames that SQLite discards on reopen — so for the default crash scenario the journal preserves *nothing*, directly contradicting `DURABILITY.md:13` ("survives SIGKILL … Zero committed records are lost"). The crash tests only pass because they manually `flush()`/`COMMIT` (`tests/runtime/test_sqlite_journal.py:230,251,478`).
  - **Why it hurts robustness** — The journal is the documented single source of truth for post-mortem debugging, and it is empty for exactly the failure operators most need to investigate.
  - **Severity** high · **Effort** small.
  - **Fix** — Commit at the end of `_do_append` (`journal.py:1095`), exactly as `LibsqlJournal` already does (line 1635); under `synchronous=NORMAL` this is a buffered write, not an fsync, so the per-turn latency budget holds. Remove the manual `flush()`/`COMMIT` from the crash tests so they exercise the real path, and correct `DURABILITY.md` to state the actual cadence and worst-case loss window.

### 4. CLI bundle triage reports wrong duration and tool-call count (HIGH)

- **`debug-validation-cli-0`** — `_summarise_bundle` reads top-level `record.get("wall_ns")` and counts `record.get("name") == "ToolCallStarted"`, but exported ZIP records nest the timestamp under `timing.wall_ns` and the journal sink writes the snake_case name `tool_call_started`. So every `.zip`/`.bundle` from `session.export_debug_bundle(...)` renders `duration = n/a` and `tool_calls = 0`.
  - **Anchors** — `src/easycat/cli/debug/bundles.py:88,95,99`; `src/easycat/debug/export.py:127`; `src/easycat/session/_journal_sink.py:101`.
  - **Why it hurts robustness** — `bundles show`/`inspect` is the advertised fastest triage surface, and two headline fields are silently wrong for the common exported case (the debugger server's `_summarise_turns` reads `timing.wall_ns` correctly, so the two surfaces disagree). A misleading observability tool is a robustness regression.
  - **Severity** high · **Effort** small.
  - **Fix** — Read `wall_ns` with a fallback to `timing.wall_ns`, and match the tool-call name against `tool_call_started`. Replace the misleading test fixture (`tests/cli/test_bundles.py:101`) with records of the real export shape — or drive it through `export_debug_bundle` — and assert non-null duration and `tool_calls == 1`.

### 5. Audio-buffer corruption on backpressure and reconnect (MEDIUM)

These independently corrupt the *audio the user actually hears* under real telephony conditions.

- **`session-audio-text-0` — Outbound TTS queue uses `DROP_OLDEST`.** Session hard-codes `DropPolicy.DROP_OLDEST` (max_size 200) on the single outbound playback queue (`_session.py:317`). When the transport stalls, `put()` does `popleft(); append(chunk)` (`_bounded_queue.py:125-136`), discarding the *earliest still-unsent* speech and playing a later fragment — the listener hears the bot jump forward mid-sentence. Worse, drops happen before `record_audio_sent`, so barge-in byte accounting and the journal never reflect the discontinuity. **Severity** medium · **Effort** small. **Fix**: default to `BLOCK` (real backpressure; `await_drain` already bounds teardown) or at minimum `DROP_NEWEST` (clean tail-cut the interruption estimator already models); apply the same change at the reconnect re-creation path (`_session.py:1213-1216`) and surface the policy on `EasyConfig`/`SessionConfig`.

- **`tts-providers-0` — Reconnect replay shifts all audio by one byte.** All three WS TTS providers replay the utterance from the top after a mid-stream reconnect, but `_replay_request` never resets `TTSBase._sample_carry` (only reset in `_start_synthesis`/`_end_synthesis`, `base.py:49,62`). If the socket dropped while a 16-bit PCM sample was split across frames, the stale 1-byte remainder is prepended to the fresh stream, **mis-aligning every subsequent sample for the whole replayed utterance** — turning the "accepted audible repetition" into full-duration static. **Anchors**: `tts/base.py:80`; `cartesia_tts.py:125`, `deepgram_tts.py:96`, `elevenlabs_tts.py:314`. **Severity** medium · **Effort** small. **Fix**: add `TTSBase._reset_audio_alignment()` (`_sample_carry = b""`) and call it at the top of each `_replay_request`; per-provider regression test feeding an odd-length chunk before the simulated reconnect.

- **`stt-providers-0` — Batch STT re-emits PARTIAL/FINAL on retry.** `OpenAISTT._transcribe_streaming` emits `STTEvent`s *while* streaming, then on `httpx.TransportError`/`TimeoutException` does `sleep; continue` to retry — but the events already pushed onto `_event_queue` are not rolled back (`openai_provider.py:88`). The consumer can see duplicate PARTIAL streams and **two FINALs for one turn**; a duplicated/late FINAL can trigger a second agent run. The streaming providers guard this with `_dropping_pending_final`; the batch provider has no equivalent. **Severity** medium · **Effort** small. **Fix**: buffer events per attempt and flush only on success (discard on retry), or track last-emitted FINAL/PARTIAL text and suppress identical re-emission.

### 6. AEC / noise-reduction degraded paths fail silently or wrongly (MEDIUM)

The audio-preprocessing chain has several places where degraded conditions corrupt the filter state or are mislabeled.

- **`vad-noise-echo-1` — Passthrough fallback crashes the happy path.** `create_noise_reducer()` auto mode returns `PassthroughNoiseReducer` (`noise_reduction.py:281`) when neither Krisp nor RNNoise is installed, but `Session.__init__` *rejects* a passthrough noise reducer with a hard `ValueError` when `enable_noise_reduction` is true (`_session.py:223-232`). So `EasyConfig(enable_noise_reduction=True)` without the optional backend crashes at construction — the factory's graceful-degradation philosophy directly contradicts the Session's strict-noop guard. **Severity** medium · **Effort** small. **Fix**: add a `fallback_policy` to `NoiseReducerConfig` (mirroring `create_echo_canceller`) so auto-mode landing on passthrough warns-and-continues with an actionable "install easycat[rnnoise]" hint instead of a hard crash; at minimum make the `ValueError` text name the missing extra.

- **`vad-noise-echo-2` — AEC sample-rate mismatch swallowed and mislabeled.** `LiveKitAEC._check_stream_rate()` raises `ValueError` on any near/far rate mismatch from `feed_reference()` (`echo_cancellation.py:97`), but `_drain_outbound_audio` wraps `_handle_audio_delivery` in `except Exception: logger.exception("Failed to send audio to transport")` (`_audio_router.py:517,521`). Since near/far streams routinely differ (8k mic / 24k TTS), the first mismatch permanently disables AEC reference for the session, **echo is never cancelled**, the `AudioOut` journal event and playback accounting are skipped, and the operator sees a misleading transport error. **Severity** medium · **Effort** small. **Fix**: give `feed_reference` its own try/except (resample the far-end chunk to the established AEC rate using existing `resample_chunk`, log once as an AEC config error), and ensure the `AudioOut` emit + byte accounting are not skipped when it raises.

- **`vad-noise-echo-4` / `vad-noise-echo-6` — Stateful filters get mid-stream silence injection.** Both `RNNoiseReducer.process` (`noise_reduction.py:93`) and `LiveKitAEC.process` (`echo_cancellation.py:126`) zero-pad the final partial frame of *every* chunk and keep no remainder buffer, unlike the VAD backends which accumulate `self._buffer` and defer the remainder (`silero.py:223`). Zero-padding a stateful recurrent denoiser / AEC3 adaptive filter injects silence into the middle of the stream, polluting GRU/filter state and degrading cancellation precisely during barge-in. **Severity** medium (RNNoise) / low (AEC) · **Effort** small each. **Fix**: add a per-stream byte remainder buffer mirroring the VAD pattern (separate near/far buffers for AEC), process only whole frames, and zero-pad only on explicit flush/close.

### 7. WebRTC/WebTransport teardown and backpressure races (MEDIUM)

- **`transports-0` — `WebRTCTransport.disconnect()` does not hold `_offer_lock`.** `_handle_offer` serializes peer replacement under `async with self._offer_lock` (`webrtc.py:519`), but `disconnect()` (line 446) takes no lock and awaits `_pc.close()`/`_site.stop()`. A client POSTing `/offer` concurrently with teardown can rebuild `_pc`, restart `_consume_audio`, and set `_client_connected` *after* disconnect nulled them — leaving a **leaked peer connection + running task that report "connected" after teardown**. `aiohttp`'s `_site.stop()` only stops *accepting* connections; in-flight handlers keep running. **Severity** medium · **Effort** small. **Fix**: acquire `_offer_lock` in `disconnect()` *and* re-check `self._connected` in `_handle_offer_locked` after its await points (discard the built PC, return 409/503 when disconnected) — locking alone is insufficient because a mid-negotiation handler still leaks its PC.

- **`transports-1` — WebTransport backpressure silently disables itself.** `_await_outbound_capacity` reaches into aioquic privates (`_quic`, `_streams`, `sender._buffer`, `webtransport.py:749,756,760`), and **every defensive miss path `return`s — i.e. reports NO backpressure**. This gate is the only documented memory bound (`_OUTBOUND_SEND_BUFFER_HIGH_WATER`); if a future aioquic release renames any of those privates, backpressure becomes a silent no-op and the exact unbounded-memory failure it prevents returns with no log/event/test signal. **Severity** medium · **Effort** small. **Fix**: probe the private accessors once at init and emit a degraded event (and log) if any are absent, so the no-backpressure fallback is observable rather than silent.

### 8. Duplicated safety-critical control flow that can silently diverge (MEDIUM)

The barge-in / turn-lifecycle paths have several hand-copied implementations of the same protocol. The risk is uniform: a fix to one copy silently skips the others, and some have **already drifted**.

- **`agent-integrations-0` — `apply_interruption` four-step atomic-write protocol copy-pasted across all 7 bridges** (~336 LOC; `openai_agents.py:244` + 6 siblings). Only `_plan_interruption`/`_apply_planned_mutation`/`_serialize_framework_state` differ per bridge; the journal-ordering contract (which `base.py` explicitly names) is duplicated. **Severity** medium · **Effort** medium. **Fix**: hoist into one shared free function in `base.py` (`run_atomic_interruption(plan_fn, apply_fn, serialize_fn, ...)`) so the contract is single-sourced and unit-testable once.

- **`session-audio-text-1` — `TTSScheduler.synthesize()` is dead but re-implements the bot-speaking start/stop/gating decision tree** that `_turn_runner._process_tts` independently re-implements (`_tts_scheduler.py:218` vs `_turn_runner.py:331-413`). Two parallel copies of the most safety-critical "when to start/stop bot-speaking and clear under the gate" logic means barge-in/clear semantics can silently diverge — and `finalize_speaking_turn` (whose docstring says it exists *so the semantics cannot diverge*) is not called by the live per-chunk path. **Severity** medium · **Effort** small. **Fix**: delete the dead `TTSScheduler.synthesize` (and its dedicated tests) OR refactor `_process_tts` to call it so the gating state machine lives in one place. No active runtime bug (the duplicate never executes), but it is a turn-lifecycle audit hazard.

- **`session-core-3` — `_execute_text_turn` duplicates `consume_agent_stream`'s bridge-event dispatch** (`_turn_runner.py:581` vs `_streaming.py:88`) and **has already drifted**: the text copy omits `cursor_entered`/`cursor_exited`/`handoff`/`state_snapshot` handling and the cancellation tool-drain logic, giving text sessions different observability/cancellation semantics than voice for the same agent stream. **Severity** low · **Effort** small. **Fix**: extract a shared `dispatch_agent_event` helper (or call `consume_agent_stream` with a no-op TTS sink) so both paths share one translation.

- **`xc-consistency-1` / `tts-providers-2` — `_emit_provider_error` + `_emit_tasks` copy-pasted into 4 providers** (3 WS TTS + `OpenAIRealtimeSTT`) while the STT side already shares one copy on `WebSocketSTTBase`. Five copies of subtle fire-and-forget task-tracking (strong-ref to avoid GC mid-emit, no-loop guard) means error paths can silently diverge. **Severity** medium · **Effort** small. **Fix**: hoist into a focused `ProviderErrorEmitter` mixin (lighter than a full `WebSocketTTSBase`); keep provider-specific context keys at the call sites.

### 9. Provider/journal inconsistencies that make degraded state undiscoverable (MEDIUM–LOW)

- **`runtime-observability-1` — Degraded mode recorded inconsistently.** `InMemoryRingBuffer._enter_degraded` writes a `JournalDegraded` marker (`journal.py:589`), but `SqliteJournal`/`LibsqlJournal` only set a flag and bump a gauge — **no record** (`journal.py:1100,1646`). A `RunBundle` loaded fresh from a `.sqlite` file has no degradation signal, so a consumer can't tell the journal silently dropped records after a backend write failed — undermining "single source of truth" for exactly the failure case that matters. **Severity** medium · **Effort** small. **Fix**: write a best-effort `DEGRADED` marker (or persist a `degraded` key in `session_state`) in the SQL `_enter_degraded` paths; the SQLite schema already has a `DEGRADED` kind whose rehydration branch (`journal.py:1163`) is currently dead.

- **`tts-providers-5` — OpenAI TTS errors never surface a provider Error event.** Three of four TTS providers emit a structured `Error` on failure; `OpenAITTS` (the default/auto-wired provider) has no `event_bus` and its `except httpx.HTTPError` drops the error entirely when `_cancelled` (`openai_tts.py:98`). A RunBundle of an OpenAI TTS outage shows no provider Error — the happy-path provider is the hardest to debug. (Note: `TTSStage` still records a stage_error, so the journal isn't fully blind — this is about richer provider-scoped context.) **Severity** low · **Effort** small. **Fix**: add `event_bus` to `OpenAITTSConfig`, add it to the factory's `needs_event_bus` tuple, and emit a provider Error mirroring ElevenLabs.

- **`tts-providers-1` — ElevenLabs `stop()` leaks the WebSocket.** It calls `super().stop()` and closes `self._response`, but in the default WS mode `_response` is always `None` and it never calls `_close_ws()` (`elevenlabs_tts.py:341`) — contradicting Deepgram's docstring parity claim. **Severity** low · **Effort** trivial. **Fix**: call `await self._close_ws()` after the response close so the three WS providers have uniform `stop()` semantics. (Note: no active production leak since nothing currently calls provider `stop()` — consistency cleanup.)

- **`transports-2` / `transports-3` — Twilio transport pair diverges on disconnect.** `TwilioTransport.send_audio` does not clear `_ws`/`_client_connected` on `ConnectionClosed` (so `has_client`/`is_connected` keep reporting a live client and every send re-fails), while its connection-variant twin does (`twilio_media.py:295` vs `709`). `send_mark` similarly diverges: one raises `RuntimeError` + propagates `ConnectionClosed`, the other soft-returns the mark name (`twilio_media.py:307,321` vs `725,731`). Two transports for the same capability obey different contracts depending on which constructor the app used. **Severity** low · **Effort** trivial. **Fix**: make both follow the soft-no-op + guarded-send pattern (or designate the receive-loop `finally` as the single source of truth and remove the eager clears).

- **`telephony-4` — Classification gate stays closed after CLASSIFYING→VOICEMAIL.** `discard()` clears the buffer but leaves `_closed = True` and the `TTSAudio` subscription active (`call_state.py:494`); the gate only reopens on transition *to* HUMAN. So leave-message TTS for `VoicemailPolicy.LEAVE_MESSAGE` is buffered and silently dropped, leaving an empty voicemail. (Latent: framework ships no internal leave-message consumer and all gated framework TTS uses `bypass_gate`.) **Severity** low · **Effort** small. **Fix**: have `discard()` fully stop the gate (`_closed=False` + unsubscribe) once buffering is no longer needed, and/or add a VOICEMAIL reopen path.

### 10. Hygiene: cancellation paths leak iterators/tasks; teardown swallows errors (LOW)

A cluster of low-severity lifecycle/hygiene issues. The existing cooperative-cancel design already makes most of these *safe* (read-only journal no-ops, bridge `finally` blocks close upstream connections), so these are robustness-in-depth, not active leaks.

- **`xc-robustness-2` / `xc-robustness-3`** — Neither `with_tts_timeout` (`timeouts.py:126`) nor `consume_agent_stream` (`_streaming.py:88`) calls `await ...aclose()` on the wrapped iterator when the consumer `break`s on barge-in, deferring provider stream finalization to GC. **Fix**: add `with contextlib.suppress(Exception): await iter.aclose()` in the `finally`. Trivial each.
- **`xc-robustness-1`** — Transport/provider fire-and-forget `_emit_tasks` are never drained/cancelled on disconnect/close (`transports/_base.py:96,236`; `elevenlabs_tts.py:356`), so a late emit can fire after `_destroy()` swaps the journal to read-only. Safe today (read-only no-op + emit try/except) but produces "Task was destroyed but it is pending" warnings. **Fix**: drain with a short timeout in teardown.
- **`xc-dead-code-7`** — Agent teardown in `stop()` swallows all exceptions with `except Exception: pass` and **no logging** (`_session.py:1392`), unlike the sibling `_close_audio_providers` which logs at debug. Teardown is exactly where leak bugs surface; silent swallowing makes them undiagnosable. **Fix**: one-line change to `logger.debug("Error closing agent during stop", exc_info=True)`.
- **`agent-integrations-1`** — `apply_interruption` step-4b success block (`record_state_snapshot` + `record_cancellation_boundary`) is unprotected (`openai_agents.py:288` + 6 siblings); a recorder failure after the mutation applied leaves the journal reading as an aborted interruption that actually completed. Low impact (no production Session passes a recorder today). **Fix**: wrap step 4b in try/except-log-continue; ideally fold into the shared orchestration helper from finding 8.

### 11. Cross-platform & onboarding robustness (MEDIUM–LOW)

- **`xc-ease-of-use-0` — `run()` crashes on Windows.** The canonical entry point unconditionally calls `loop.add_signal_handler(SIGINT/SIGTERM, ...)` (`helpers.py:114`, repeated at `:33`), which raises `NotImplementedError` on the default Windows `ProactorEventLoop`. The single most-advertised onboarding call (`run(EasyConfig.mic(agent=...))`, README:112) raises before the bot starts, with an asyncio-internals traceback. **Severity** medium · **Effort** small. **Fix**: wrap each `add_signal_handler` in `try/except NotImplementedError`, falling back to `signal.signal` (main thread) or KeyboardInterrupt-driven shutdown so teardown still runs via `__aexit__`.
- **`xc-ease-of-use-1` — `_resolve_api_key` silently defaults unknown providers to `OPENAI_API_KEY`.** `quick.py`'s hand-maintained map omits `cartesia` and falls through to `OPENAI_API_KEY` (`quick.py:26-31,37`), a third drifted copy of the provider→env-var map. `transcribe_file(..., provider="cartesia")` resolves the wrong key and fails with an opaque 401. **Severity** low · **Effort** trivial. **Fix**: delete the local map and resolve from the drift-guarded factory catalogs, raising a clear unknown-provider error instead of falling through.
- **`xc-consistency-3` — `needs_event_bus` lists are hand-maintained and diverge across the STT/TTS factories** (`stt/factory.py:98`, `tts/factory.py:122`). Forgetting to append a new event-bus-aware provider means its journal `Error` events silently never fire — invisible until a blank bundle. **Severity** low · **Effort** small. **Fix**: derive structurally — `if any(f.name == "event_bus" for f in fields(config)) and config.event_bus is None: replace(config, event_bus=...)`.

### 12. Encapsulation leaks that make the architecture fragile under change (LOW)

These don't corrupt turns today but couple layers to private internals, so any rename breaks robustness silently past the type checker.

- **`stt-providers-3`** — `STTCommitter.commit_now` reads `getattr(stt, "_bytes_since_last_commit", None)` (`_stt_committer.py:208`), a private field only `OpenAIRealtimeSTT` defines, so the journal records `None` for every other provider. **Fix**: add a `@runtime_checkable PendingCommitReporter` capability protocol.
- **`turn-management-1` / `xc-architecture-4`** — Session reaches `self._turn_manager._config.endpoint_detector # type: ignore[attr-defined]` in four spots (`_session.py:400,548`), creating two sources of truth for the detector. **Fix**: add a public `endpoint_detector` property on `TurnManager`; trivial.
- **`config-public-api-7` / `agent-integrations-3`** — `_inject_agent_runtime` mutates bridge privates (`_mcp_servers`/`_model`/`_api_key`) by name guarded by `hasattr` (`config.py:413`), so a conformant custom bridge silently gets no MCP wiring and no warning. **Fix**: thread these through bridge construction (built-ins already accept them) or add an optional `configure_runtime(...)` protocol method.
- **`xc-consistency-5`** — `TTSProvider`/`STTProvider` protocols omit `close()`/`aclose()` even though `Session._close_audio_providers` relies on it for teardown (`providers.py:80`), so a third-party provider author has no signal a teardown hook exists → leaked httpx clients/sockets. **Fix**: document an optional `aclose()` hook in the protocol docstring.

# 4. Understandability

## Understandability

The framework works on this axis but is dragged down by a recurring pattern: **safety-critical logic (turn lifecycle, interruption, journaling) gets hand-copied across siblings, then quietly diverges.** A reader auditing any one path cannot tell which copy is live or whether the copies still agree. The second-largest theme is **god-objects/god-functions** (`Session`, `Session.__init__`, `create_session`, `config.py`, `journal.py`, `LangGraphBridge.invoke`) where the easy path is buried under advanced wiring. The cheapest wins are **dead code and stale docs** that actively mislead.

### Theme 1 — Duplicated safety-critical logic that has already diverged (highest impact)

These are the issues where two copies don't just cost double maintenance — they have *already* drifted, giving the same operation different behavior depending on the path.

- **`apply_interruption` orchestration copy-pasted verbatim across all 7 agent bridges.** `integrations/agents/openai_agents.py:244` + 6 siblings (`pydantic_ai.py:167`, `responses_api.py:269`, `generic_workflow.py:203`, `langchain.py:404`, `langgraph.py:647`, `llama_agents.py:226`, `base.py:339`). ~336 LOC of identical "four-step atomic write ordering" — the documented barge-in contract over the journal. Seven copies of an atomic-write protocol means any ordering/error-handling fix must land in seven places. **Severity: medium · Effort: medium.** Hoist into one shared free function in `base.py` (`run_atomic_interruption(plan_fn, apply_fn, serialize_fn, ...)`); each bridge passes only its three bridge-specific callables. Makes it unit-testable once.

- **Two divergent implementations of bridge-event → EasyCat-event translation.** `session/_turn_runner.py:581` (`_execute_text_turn`) hand-rolls the same per-kind switch (`done`/`text_delta`/`tool_started`/…) that `consume_agent_stream` already implements at `session/_streaming.py:88`. **They have already drifted:** the text copy omits `cursor_entered`/`cursor_exited`/`handoff`/`state_snapshot`, first-token-latency capture, and the cancellation tool-drain logic. Text sessions get different observability/cancellation semantics than voice for the identical agent stream. **Severity: low · Effort: small.** Extract a shared `dispatch_agent_event` helper (or have `_execute_text_turn` reuse `consume_agent_stream` with a no-op tts sink) so both paths share one translation.

- **Dead-but-elaborate duplicate of the bot-speaking start/stop gating.** `session/_tts_scheduler.py:218` (`TTSScheduler.synthesize`) has no production caller — every live path goes through `_turn_runner._process_tts` (`_turn_runner.py:331-413`), which re-implements the same `bot_started_speaking` + `finalize_speaking_turn` + `reset()` decision tree (`_tts_scheduler.py:243-277` vs `_turn_runner.py:358-413`). The `finalize_speaking_turn` docstring claims it exists "so the barge-in / clear semantics cannot diverge" — but the live path doesn't call it per-chunk. A lifecycle auditor must reverse-engineer which copy runs. **Severity: medium · Effort: small.** Prefer deleting `TTSScheduler.synthesize` and its synthesize-only tests; or, if single-shot synthesis is coming, route `_process_tts` through it so the gating state machine lives in one place.

- **STT/TTS WebSocket error-emission copy-pasted 4–5 ways with no shared base.** `_emit_provider_error` + the GC-safe `_emit_tasks` task-tracking is reimplemented near-verbatim in `tts/cartesia_tts.py:263`, `tts/deepgram_tts.py:244`, `tts/elevenlabs_tts.py:363`, and `stt/openai_realtime_provider.py:432` — while the STT side already proved the abstraction in `stt/websocket_base.py:187`. The copies differ only in a hardcoded provider string, and divergence is already visible (only ElevenLabs attaches `http_status`/`body`; only Deepgram documents `ws_close_code`). **Severity: medium · Effort: small.** Hoist `_emit_provider_error` + `_emit_tasks` into a small `ProviderErrorEmitter` mixin parameterized by stage + provider name; reconcile `OpenAIRealtimeSTT`'s `*, code, buffer_bytes` signature to `**context` first. *(Merges `tts-providers-2`, `xc-consistency-1`.)*

- **`OpenAIRealtimeSTT` bypasses `WebSocketSTTBase` and re-rolls ~120 LOC of WS lifecycle.** `stt/openai_realtime_provider.py:79` extends bare `STTBase` while Deepgram/Cartesia/ElevenLabs all derive from `WebSocketSTTBase` (`stt/websocket_base.py:30`). It hand-rolls `_receive_loop`, `_close_connection`, `_log_close_task_exception`, a private `_emit_tasks`, and `_emit_provider_error`; its `_close_connection` uses a hardcoded 2.0s timeout vs the base's configurable `_close_timeout`. Base fixes (drain/reconnect) silently skip this provider. **Severity: medium · Effort: medium.** Reparent onto `WebSocketSTTBase` using the ElevenLabs provider as template; keep only the genuinely OpenAI-specific bits (`session.update` handshake, the `_session_ready` 5s gate, commit-minimum logic). Rename `_handle_message` → `_handle_json_message` and update the 4 test call sites. *(Merges `stt-providers-1`, `xc-consistency-4`.)*

### Theme 2 — God-objects and god-functions hide the easy path

- **`Session` is a ~1822-LOC god-object; `Session.__init__` is a 386-line god-constructor.** `session/_session.py:158` (class), `:171-555` (`__init__`). ~90 methods spanning construction, lifecycle, telephony accessors, caller-id rendering (`:1036`), opt-out detection (`:1670`), session-action draining (`:1562`), and journal backend swapping (`:1431`). The journal is documented as "the single source of truth for all observability," yet a schema change must be applied in 2-3 places and a libSQL-only divergence is invisible in SQLite-only tests. **Severity: medium · Effort: small–medium.** Extract a `_SqlJournalBase` (or at minimum a shared `_JOURNAL_INSERT_SQL` constant + `_encode_journal_row(...)` builder) holding the append guard/timing, INSERT body, read/slice, `_enter_degraded`, and `_row_to_record`; subclass for connection/commit/sync specifics. Add a unit test round-tripping `LibsqlJournal` under an in-process shim. *(Merges `runtime-observability-2`, `xc-architecture-2`.)*

### Theme 3 — Boundary leaks: orchestrator reaches into private collaborator internals

A cluster of `getattr`/private-attribute reach-ins, several muted with `# type: ignore[attr-defined]`, that defeat the documented `Session → Stages → Providers` layering and break silently past the type checker on rename.

- **`Session._execute_text_turn` wraps a private `_turn_runner._execute_text_turn`; the wrapper is itself unused.** `session/_session.py:1772-1773`. **Severity: low · Effort: trivial.** Delete the dead `Session._execute_text_turn` wrapper; if kept for symmetry, promote the runner method to public `execute_text_turn`.

- **`STTCommitter` reads provider-private `_bytes_since_last_commit` via `getattr`, off the Protocol.** `session/_stt_committer.py:208`. # 5. Architecture

I have all the verified findings I need in the JSON. Let me write the architecture section directly.

## Architecture

The framework is fundamentally well-layered (`Session → Stages → Providers`, Protocol-over-inheritance, factory-driven wiring), and none of the findings below are correctness bugs — every one is a maintainability, understandability, or robustness concern. They cluster into five themes. Within each, items are ordered by leverage.

### 1. God-objects and god-functions: the `Session` and `config.py` hubs

Two files concentrate almost all assembly complexity, and they are the first things any contributor or adopter must read.

**`Session` (1822 LOC, ~90 methods) — `src/easycat/session/_session.py:158`** owns construction, lifecycle, properties, telephony accessors, opt-out, caller-id rendering, session-action draining, and journal-backend swapping. The single densest region is the **386-line `__init__` (`_session.py:171–555`)**, which instantiates 7 stages, `AudioRouter`, `STTCommitter`, `TTSScheduler`, `CancelOrchestrator`, and `TurnRunner` inline, threading ~15–20 forward-referencing lambdas between them (e.g. `is_stt_active=lambda: self._stt_committer.is_active` at `_session.py:429` references a collaborator not built until `:436` — safe only because the lambda is deferred). Wiring order is load-bearing and documented only by prose comments (`"Must be constructed after all 7 stages"`, `:484`). (Findings `session-core-2`, `xc-architecture-1`.)

- *Recommendation (medium effort, medium priority):* Extract the collaborator graph into `session/_builder.py` (`_SessionWiring`) that takes config+providers and returns an assembled bundle, leaving `__init__` a short field-assignment shell. The highest-value sub-part is replacing the forward-reference lambdas with an explicit typed wiring context so init order becomes type-checkable rather than comment-enforced. Then move the self-contained opt-out + greeting + caller-id trio (~120 lines, e.g. `_session.py:1036`, `:1670`) into a `GreetingController` / opt-out policy collaborator that subscribes to the event bus itself, and optionally collapse the 6 telephony delegating properties behind one `session.telephony` facade. Leave `stop()` / `cancel_turn` / lifecycle on `Session` — that is its legitimate job. Treat an auditable `stop()` ordering and a sub-minute-scannable constructor as the goal, not the line count.

**`config.py` (1593 LOC, 24 top-level functions) — `src/easycat/config.py`** conflates the public `EasyConfig` / `TextSessionConfig` dataclasses (`:505`, `:1089`) and the happy-path STT/TTS/VAD wiring with deep outbound-telephony/IVR/screening machinery (`_OutboundPipelineWiring` `:1261`, `_wire_outbound_pipeline` `:1324`, `_create_outbound_helpers` `:1425`), the `create_session` factory (`:795–977`), debugger-UI launching (`_maybe_launch_debugger_ui` `:1037`), and a `record_to` monkeypatch hook. The factory itself is a god-function: `create_session` inlines provider construction, agent adaptation, telephony assembly, a nested `audio_gate()`, IVR navigator + 4 event-bus subscriptions, Twilio/outbound identity stamping, and debugger auto-launch. (Findings `config-public-api-2`, `xc-architecture-6`, `xc-ease-of-use-5`.)

- *Recommendation (medium effort, low–medium priority):* Extract the outbound/telephony runtime wiring (`_OutboundPipelineWiring`, `_wire_outbound_pipeline`, `_create_telephony_helpers`, `_create_outbound_helpers`, `_run_agent_once`, plus the ~22 heavy `easycat.telephony.*` runtime imports) into a dedicated module (e.g. `config/_telephony_wiring.py`) imported lazily inside `_create_telephony_helpers` only when `config.telephony` is set. This shrinks the core config module a newcomer reads (the genuine win — telephony pulls no heavy deps, so do not justify it on cold-start). Because tests import `_create_telephony_helpers` from `easycat.config` directly, keep a thin lazy-delegating wrapper there, and keep `EasyConfig` / `create_session` / `create_text_session` re-exported from the package root.

### 2. Eager-import coupling defeats the lazy-load design

**`src/easycat/config.py:18–96`** has ~80 module-level imports pulling in the whole telephony subsystem, every transport, and every STT/TTS provider config. But `__init__.py` (`:14`, `:37`) advertises PEP 562 lazy loading "so cold starts stay cheap," and `EasyConfig` / `create_session` are registered for lazy loading. Touching `EasyConfig` imports `config.py`, which imports everything: `python -X importtime -c 'import easycat; easycat.EasyConfig'` pulls in `easycat.telephony` (66ms), `easycat.session._session` (166ms), httpx, and all providers — so the most common first call, `EasyConfig.mic(...)`, drags in machinery a local-mic developer never uses, defeating the stated goal. (Finding `config-public-api-1`.)

- *Recommendation (medium effort, medium priority):* Defer the heavy/optional imports into the functions that use them, following the existing `_inject_agent_runtime`/catalog lazy pattern in the same file. Move telephony-runtime and transport-implementation imports into `_create_transport` / `_create_telephony_helpers` / `_create_outbound_helpers` / `_create_action_executors`; replace the module-level `_TRANSPORT_FACTORIES` dict with a function that builds the mapping lazily; ensure config dataclasses used as `default_factory` (`DTMFAggregatorConfig`, `VoicemailDetectorConfig`, `LocalTransportConfig`) come from lightweight config-only modules so importing them does not trigger `telephony/__init__.py`'s eager fan-out. Add a regression test asserting that accessing `easycat.EasyConfig` / constructing `EasyConfig.mic(...)` does **not** import `easycat.telephony.*` or unused transport modules. (This naturally complements the §1 telephony-wiring extraction.)

### 3. Leaky abstractions: reaching across boundaries

A recurring pattern: a higher layer pokes private attributes of a lower one, muting the type checker with `# type: ignore[attr-defined]` or silently no-op'ing via `getattr`/`hasattr`. These defeat the layering the codebase otherwise enforces.

**Session → TurnManager private reach-ins (`_session.py:399–404`, `:548–554`; `turn_manager.py:161`).** Session reads `self._turn_manager._config.endpoint_detector # type: ignore[attr-defined]` in two places (with dead `_config is not None` guards) to decide endpoint-stage binding. TurnManager has no public accessor and already stores `self._endpoint_detector` (`turn_manager.py:161`), so the detector lives in two places. `Session._execute_text_turn` (`_session.py:1773`) also calls the private `self._turn_runner._execute_text_turn(...)` — and nothing calls that wrapper. (Findings `turn-management-1`, `xc-architecture-4`.)

- *Recommendation (trivial effort, do first — highest leverage-per-effort in this section):* Add `@property def endpoint_detector` to TurnManager returning `self._endpoint_detector`; use it in both wiring blocks and delete the four `# type: ignore` reach-ins and the dead guards (collapsing the two sources of truth). Delete the unused `Session._execute_text_turn` wrapper entirely.

**Session → STT-provider private field (`_stt_committer.py:208`).** `commit_now()` does `getattr(self._stt_getter(), '_bytes_since_last_commit', None)`. Only `OpenAIRealtimeSTT` defines that field (`openai_realtime_provider.py:107`); the `STTProvider` Protocol (`providers.py:42`) exposes nothing, so the journal silently records `None` for every other provider. (Finding `stt-providers-3`.)

- *Recommendation (small effort):* Add a `@runtime_checkable` capability Protocol `PendingCommitReporter` with `pending_commit_bytes() -> int | None` (mirroring `VersionedProvider`, `providers.py:24–36`), implement it on `OpenAIRealtimeSTT`, and in `commit_now()` use `provider.pending_commit_bytes() if isinstance(provider, PendingCommitReporter) else None`. Makes the coupling discoverable, type-checkable, and rename-safe while preserving the `None` fallback.

**Config/factory → agent-bridge private fields (`config.py:413–420`; `_factory.py:60,63`; `responses_api.py:78`).** `_inject_agent_runtime` mutates `inner._mcp_servers`, `inner._model`, `inner._api_key` by name (reaching through `agent._agent`), and `auto_adapt_agent` sets `agent._agent` / `agent._is_bridge`. None are part of the `ExternalAgentBridge` Protocol. Only the OpenAI/Pydantic bridges define `_mcp_servers`, so the `hasattr` guard silently no-ops MCP injection for any other (including user-authored) bridge — a config setting dropped with no diagnostic. (Findings `config-public-api-7`, `agent-integrations-3` — same root cause.)

- *Recommendation (small–medium effort, low priority — built-ins are uniformly covered today):* Thread `mcp_servers`/`model`/`api_key` through bridge **construction** in `auto_adapt_agent` (the built-in constructors already accept `mcp_servers=`), eliminating the name-based probe for built-ins. For custom bridges, add an optional Protocol hook `configure_runtime(*, mcp_servers=(), model=None, api_key=None)` with a default no-op so injection targets a declared surface. Also consider collapsing the two MCP channels into one by having bridges read `recorder.context.mcp_servers` at invoke time and dropping the `_mcp_servers` mutation entirely.

**TurnRunner bypasses its own AgentStage (`_turn_runner.py:489,502,550`; `_session.py:392`).** TurnRunner correctly streams through `self._agent_stage.execute_streaming(...)` (journal-recorded), but reaches around the stage to the live agent for `replace_last_assistant_text` and interruption handling (`self._agent().…`). So agent streaming is recorded while the interruption/rewrite mutations on the same agent are not — corrupting the debug record of exactly the hardest-to-debug path (barge-in). (Finding `xc-architecture-7`.)

- *Recommendation (small effort):* Pick one rule. Preferred (matches the documented "stages are the recording boundary"): add thin `apply_interruption` / `replace_last_assistant_text` wrappers on `AgentStage` that journal and delegate, and call those from TurnRunner. Alternatively thread a `JournalAgentRecorder` through the interruption path. Document that `assistant_interruption_notified` + `markdown_stripped` already cover the high-level facts, so the gap is limited to framework-state atomicity records.

### 4. Cross-subsystem duplication

**Three journal backends duplicate their core verbatim — `src/easycat/runtime/journal.py` (1826 LOC, 8 classes, no SQL base).** `SqliteJournal`, `LibsqlJournal`, and `InMemoryRingBuffer` each carry a near-identical `append()` wrapper (degraded/closed guard + `try/_do_append` + histogram timing): `:861–904`, `:1454–1497`, `:406–449`. - *Recommendation (medium effort, medium priority):* Extract a `_SqlJournalBase` holding the `append()` guard+timing wrapper, the `_do_append` INSERT body, `read()`/`slice()`, `_enter_degraded`, and the static `_row_to_record`; have `SqliteJournal` and `LibsqlJournal` subclass it and override only connection/commit/sync specifics (savepoint/clean-close vs. commit-per-append/sync). `ReadonlySqliteJournal` can call the base too. For the append wrapper shared with `InMemoryRingBuffer`, factor a module-level `_timed_append` helper rather than forcing the ring buffer under the SQL base (its `_do_append`/`_enter_degraded` semantics differ). While doing this, deliberately resolve whether SQL backends should also persist a `JournalDegraded` marker rather than leaving it an accidental copy-paste divergence.

### 5. Protocol/implementation drift: the WebSocket-STT outlier

Base fixes (drain/reconnect) silently skip this provider. (Findings `stt-providers-1`, `xc-consistency-4` — same item.)

Replace plumbing with `super().__init__(provider_name="openai_realtime_stt", provider_error_name="openai-realtime", close_timeout=2.0)`; in `_on_start` call `_connect_websocket(...)`; rename `_handle_message` → `_handle_json_message`; delete `_receive_loop`, `_close_connection`, `_log_close_task_exception`, the private `_emit_tasks`, and `_emit_provider_error`. **Preserve the genuinely OpenAI-specific bits** — the `session.update` handshake (`_send_session_update`), the `_session_ready` future + its 5s wait (no base equivalent; add a narrow post-connect "ready gate" the base awaits), the 100ms/4800-byte commit-minimum logic, and the structured error context (`code`, `buffer_bytes`) threaded through the base `_emit_provider_error(exc, **context)`. Update `tests/stt/test_stt_openai_realtime.py`, which calls `_handle_message` at 4 sites (502, 527, 534, 540).

**`TTSMarkers` carries mutually-incompatible provider-native shapes no consumer reads — `src/easycat/tts/base.py:92`.** `_make_markers_event(markers: list[dict])` is typed `list[dict]`, but Cartesia yields its `word_timestamps` object (`cartesia_tts.py:209`) and ElevenLabs yields a structurally different char-level `alignment` dict (`elevenlabs_tts.py:214`); Deepgram emits none. The only handler stores them opaquely (`_journal_sink.py:229`), and no code inspects the contents — a uniform event type that means two different things per provider and is effectively dead, untyped payload. (Finding `tts-providers-3`.)

- *Recommendation (small effort, low priority — reasonable to defer/batch):* Either (1) introduce a documented typed marker schema (a frozen dataclass mirroring the STT-side `WordTimestamp`) and normalize Cartesia's `word_timestamps` and ElevenLabs' `alignment` into it at the emit sites, tightening `_make_markers_event` off bare `list[dict]`; or (2) if no consumer is planned, delete the two emit sites, the synthesizer's MARKERS branch (`_tts_synthesizer.py:167–173`), and the journal handler, retiring the event until a real cross-provider contract exists. Either way add a one-line note near the `TTSMarkers` mention in `providers.py`.

### 6. Factory/Session contract contradiction (robustness)

**`create_noise_reducer` passthrough fallback vs. Session's noop-rejection guard.** `create_noise_reducer()` auto mode returns `PassthroughNoiseReducer()` (`noise_reduction.py:281`) when neither Krisp nor RNNoise is installed, and that passthrough sets `is_passthrough_provider = True` (`:234`). But `Session.__init__` rejects a passthrough noise reducer when noise reduction is enabled with a hard `ValueError` (`_session.py:223–232`), and `config.py:820` builds the reducer whenever `enable_noise_reduction` is true. So `EasyConfig(enable_noise_reduction=True)` on a machine without the optional backend **crashes at construction** — the factory's graceful-degradation philosophy directly contradicts Session's strict-noop philosophy, the opposite of "easy to use, sensible defaults." (Finding `vad-noise-echo-1`.)

- *Recommendation (small effort, medium priority — the only finding with a user-facing failure):* Align the noise-reduction chain with the echo-cancellation contract (`echo_cancellation.py:226–234`). Add a `fallback_policy` to `NoiseReducerConfig` (default `"passthrough"` that logs an actionable warning when auto-mode lands on passthrough; `"error"` to fail loudly with an "install `easycat[rnnoise]`" message). Then either make Session treat explicit-enable + passthrough as a warning, or ensure the factory does not return a passthrough into an `enable_noise_reduction=True` session without first logging the hint. At minimum, fix the opaque `ValueError` text (`_session.py:230–232`) to name the missing optional extra.

### 7. Oversized bridge methods (low priority)

**`LangGraphBridge` is a single 1569-line class with a 284-line `invoke()` — `src/easycat/integrations/agents/langgraph.py:95`** (`invoke` `:341–625`); `LangChainBridge.invoke` is 187 lines (`langchain.py:207`). By contrast `PydanticAIBridge.invoke` is 13 lines and `LlamaAgentsBridge.invoke` is 53 — the contract does not require this size. `invoke` is the most robustness-critical bridge method (streaming, cancellation, HITL, checkpoint trail, interruption), and a 284-line method with interleaved cursor lifecycle, stream-chunk handling, and checkpoint logic is where barge-in/cancel races hide. (Finding `xc-architecture-5`.)

- *Recommendation (medium effort, low priority — maintainability, not robustness):* Extract two cohesive sub-units from `LangGraphBridge.invoke`: a `_drive_stream(...)` async-generator holding the `astream_events` loop plus its `except Exception`/`except BaseException` cleanup, and a `_finalize_done(...)` for the post-stream `done.text` decision (`:572–623`); apply an analogous split to `LangChainBridge.invoke`. **Preserve the rationale comments** (`:444–462`, `:520–534` document non-obvious cancellation/commit-ordering constraints) and keep strong barge-in/cancel test coverage through the refactor.

Medium-leverage structural work: the `SessionBuilder`/`_SessionWiring` extraction and config telephony-wiring split (§1) paired with the lazy-import fix (§2, same surface), and the `_SqlJournalBase` extraction (§4). Lower-priority symmetry/cleanup, sequence behind any correctness work: the STT-provider capability Protocol and agent-bridge `configure_runtime` hook (§3), the `OpenAIRealtimeSTT` reparent and `TTSMarkers` decision (§5), and the LangGraph/LangChain `invoke` splits (§7).

The codebase already invested in the exact robustness the finding asks for, just at the provider layer (per-operation timeouts) rather than as one outer wait_for — a valid and arguably better-scoped pattern. The only residual is a narrow theoretical case: a `_connected`-set live TCP socket whose peer is wedged with no TCP backpressure could in principle block the raw `await ws.send(...)` (not wrapped in wait_for), but that is an OS-level edge, not the mid-reconnect/never-returns scenario claimed, and is far less likely than the finding implies.
- **Session pokes provider private attrs (_config/_event_bus) with reflection + bare except, duplicating the typed event-bus injection the factories already do** (`src/easycat/session/_session.py:1777`) — The cited code exists verbatim. src/easycat/session/_session.py:1777-1792 `_maybe_attach_event_bus` does `cfg = getattr(provider, "_config", None)`, then `setattr(cfg, "event_bus", self.event_bus)` and `setattr(provider, "_event_bus", self.event_bus)`, each in `try/except Exception: pass`. However the finding is substantially overstated and its core robustness premise is unreachable:  1) The central robustness claim — a provider whose `_config` is a frozen/slots dataclass silently never gets the bus, so Deepgram/ElevenLabs emit no events invisibly — does NOT hold. I checked every event-bus-needing config (DeepgramSTTConfig, ElevenLabsSTTConfig, CartesiaSTTConfig, OpenAIRealtimeSTTConfig at deepgram_provider.py:16, elevenlabs_provider.py:30, cartesia_provider.py:16, openai_realtime_provider.py:53; DeepgramTTSConfig/ElevenLabsTTSConfig/CartesiaTTSConfig at deepgram_tts.py:22, elevenlabs_tts.py:43, cartesia_tts.py:32). Every one is a plain `@dataclass` — none use frozen=True or slots=True. So `setattr(cfg, "event_bus", ...)` cannot raise for any shipping provider; the `except: pass` never swallows a real failure. Providers read the bus live (deepgram_provider.py:69 `event_bus=self._config.event_bus`, elevenlabs_tts.py:307/372), so post-construction mutation actually works.  2) The "two competing mechanisms" framing is inaccurate. They are not competing — `_maybe_attach_event_bus` guards on `getattr(cfg, "event_bus") is None` (line 1781). In the create_session path (config.py:815-816, 898) the factory already set cfg.event_bus via replace(), and the SAME bus is passed to SessionConfig(event_bus=...), so the attach is a guarded no-op for factory-built providers — no double-fire, no conflict. The reflective attach is the safety net for the direct `Session(SessionConfig(provider_instance))` path, where the factory never ran, AND for transports (called at _session.py:240) which the factories do not replace-inject at all. _create_transport (config.py:1373+) performs the identical `_event_bus is None`→assign pattern for pre-built transport instances, so the reflective approach is the established idiom here, not an outlier.  3) The primary proposed fix ("have providers declare need via the factory isinstance/replace path") does not work for the direct-Session path the attach exists to serve: by then the provider instance is already constructed and `replace()` on its config cannot reach back into it. Making injection raise rather than pass could break legitimate custom providers. Only the secondary nit (log at warning instead of pass) is defensible, and even that is cosmetic given no real provider triggers the except.  There is a small genuine readability wart (a typed registry path plus a reflective fallback for the same concern), but it is low severity: no active robustness hole, no ease-of-use impact, just minor understandability cost.
- **BoundedAudioQueue.get()/get_nowait() are not safe for concurrent consumers (check-then-popleft race) despite a general-purpose docstring** (`src/easycat/_bounded_queue.py:178`) — The finding's core claim is wrong. It asserts get() does `while self.empty(): ... wait()` then "unconditionally" popleft, so two consumers woken by one put() both reach popleft() and the second raises IndexError. Reading src/easycat/_bounded_queue.py:178-191, the `while self.empty()` loop IS the post-wake re-check: after `await self._not_empty.wait()` (line 184) a resumed waiter returns to the top of the loop and re-evaluates self.empty(). Critically, there is NO await between the loop guard and `chunk = self._queue.popleft()` (line 186), so in single-threaded asyncio the first woken consumer exits the loop and drains the one chunk atomically without yielding. When the second consumer resumes it re-evaluates `while self.empty()` — now True — and loops back to wait() instead of calling popleft(). The finding's key premise that the second consumer re-checks "self.empty() (still False)" is false; the first already drained it. I empirically confirmed this with two stress tests (5000 trials × 8 concurrent consumers woken by a single put; and 3000 trials × 10 consumers × 3 interleaved puts): 14,000 successful gets and ZERO IndexErrors or any non-cancellation exceptions. The cited asymmetry (locked put BLOCK path at lines 158-168 vs unlocked get) is justified, not a bug: the BLOCK producer awaits `_not_full.wait()` (line 146) BEFORE appending, and _not_full is set by get/get_nowait/flush/close, so multiple producers can pass the wait and need lock+re-check serialization; the get path needs no lock because its loop guard re-checks emptiness with no intervening await. The supervisor's potential second consumer (supervisor.py:146) operates on a plain asyncio.Queue, not BoundedAudioQueue. The IndexError mechanism is hallucinated.
